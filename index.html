Как мы создавали свой Framework и почему?

Константин Кривленя

Почему мы решили написать свой framework.

Пролог.

  1. Страницы на aspx c вкраплением ExtJs 3.x компонентов
  2. Сложность покрытия unit тестами клиентской части.
  3. Неудобство кастомизировать и создавать собственные компоненты
  4. Тежелый DOM. На большом количестве данных - проблемы с производителностью.

Поиск алтернатив в начале 2011.

  1. ExtJs 3.x недостатки описаны выше
  2. Angular.js и ember.js, только начинают развиваться, нет даже первых стабильных версий.
  3. Backbone.js, тоже только начало. Доволи строгая модель данных. Не очень подходит к нашей доменной области.
  4. В общем тяжелый 2011 год

Что мы хотели от framework.

Что же нужно для счастья от современного framework

  1. Слабосвязная архитектура. Маштабируемость.
  2. Легкий и удобный способ покрытия тестами.
  3. Низкий порог вхождения.

Архитектура framework

  1. Событийно-ориентированная модель
  2. Dependency injection
  3. Traits

Преимущества событийно-ориентированной модели

  1. Слабая связность компонетнов и его частей. Любая функциональность может быть преиспользована. Легко отключена или подключена.
  2. Хорошо тестируемый код. Может быть протестирована, как отдельные части системы, так и в контексте.
  3. Проверенный временем подход.

Проблемы framework основанного на модели событий, и как мы с ними боремся.

Часто необходимо было реагировать на склейку несколький событий. Придилось писать код вроде ниже приведенного: 
            bus.on('afterRender',function(event,$element){
             bus.on('settingsChanged',function(event,settings){
             /**Какой-то код**/
             })
            })

Что делать?

Пишем свой язык описания склейки событий.

Возможности языка

Постой способ подписки на события, описываем функции как property объекта 
                    "bus afterRender":function(evt,data) {
                     /**реагируем на события после рендера шаблона**/
                    }

Возможности языка

Склеиваем события, произошедшие однакратно. 
                    "bus afterRender + settingsChanged":function(evt,
                            $element,    settings) {
                        /**реагируем на события рендера шаблона и изменения настроек**/
                    }

Возможности языка

Используем данные события произошедшего отнакратно с повторяющимся событием. 
                    "bus afterRender:last + settingsChanged":function(evt,
                            $element,    settings) {
                       /**реагируем на события рендера шаблона(данные с послденего проброшеного
                        события)и изменения настроек**/
                    }

Возможности языка

Используем порядок возникновения событий. 
                    "bus settingsChanged > afterRender":function(evt) {
                       /**реагируем на события рендера шаблонаи изменения настроекучитывая порядок возниконовения**/
                    }

Возможности языка

А можно три события склеить? Да кончно можно. Даже так: 
                    "bus settingsChanged + ... +n":function(evt,data,..,nData) {
                       /**Данные получаем в аргументах**/
                    }

Еще проблемы? Да, гибкость.

Каждый extension может генерировать и реагировать на события. Поэтому делаем соглашения, события для других extensions может генеривровать либо main extension, либо model extension. В дополнительных extensions создаем точки входа, которые необходимы для его работы.
